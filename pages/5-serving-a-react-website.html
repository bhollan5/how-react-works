5. Serving a React website

React is a powerful tool for creating website interfaces.

But when it comes to a React website's server, there are several options to consider [1].

Below are, in my opinion, the two main options for modern websites:

  • A static website
  • A single page web app








Serving a static website

A static website is a website where pages are served to browsers exactly the way they're stored on the server [2].

Static websites are generally easy to serve. A service like NginX can be used to map URL routes to HTML pages in a file tree, and load other assets in a simlar way.


You can use ReactJS with a static website -- just link the React scripts in your code, and you can use React tools in whatever JavaScript you add.

You can also use JSX to build a static website, with Babel.


React's official website recommends[3] the service Gatsby. Gatsby is a framework that provides all the React features, and also bundles your code with Webpack, so it will run faster in the browser.

Gatsby also provides templated React components and page layouts for you to use, and other features, like GraphQL set up.









Serving a single page web app

A single page app, or SPA, is a website design pattern where the website loads a single HTML file, and any other page route content is then loaded into that HTML file, without actually reloading the page [4].


This set up has a few advantages. For one thing, it's easy to reuse the same website header and navigation links, while still having many different pages with different content.

A SPA website doesn't reload completely between page route changes, which means transitions between pages can be faster and smoother.


You'll hear a lot about server-side rendering versus client-side rendering with SPAs. With server-side rendering, an SPA will load the entire website, including the route page's contents, the first time the page loads.

Server-side rendering is better than client-side rendering, where the base page is sent to the browser, which then makes an additional call to the server to load the route's content.

However, once the website is initially loaded, when the user navigates to a new page, both strategies will load the route page's content from the server, and render it client-side. That's simply how SPAs work.


React's website recommends[5] NextJS for building single-page apps, a framework for server-side rendering.

NextJS is built on NodeJS, and provides "styling and routing solutions".


I should mention that this website is a client-side rendered SPA app that doesn't use NextJS.

I'm using my own simple NodeJS set up to serve the initial index.html file, and then a browser script to make calls back to the server to load individual route pages.
